# 張り合い・未決事項

[← 目次](./README.md)

## Pick 文化 vs 一貫性

全世界から pick すると見た目がバラける。**公開 primitive は少数に絞り**、ドメイン block はプロダクト registry でトーンを統一するのが現実的（→ [03-product-registry.md](./03-product-registry.md)）。

## copy-in モデルと公認のドリフト

`shadcn add` 後にローカルで改変したり、AI が参考 block を直書き fork すると「公認から外れた fork」が増える（→ [07-ai-enablement.md](./07-ai-enablement.md)）。

対処の方向性：

- 憲法は **プロダクト registry の URL** を正とし、トークン・公認一覧を参照する
- 変更はプロダクト registry に PR → 再公認、が文化になる
- 厳しめなら「本番は公認 block のみ」「カスタムはラップ層で」

**アプリ repo は消費者、正はプロダクト registry repo** に寄せると、お墨付きモデル（→ [06-governance.md](./06-governance.md)）と相性がよい。

**未決：** 公認 block をアプリが使うとき、コードは repo に copy するか、registry を常に参照するか。

[08-stakeholder-workflow.md](./08-stakeholder-workflow.md) / [06-governance.md](./06-governance.md) で段階は定義済み。未決なのは運用の細部：

- デザイナー vs エンジニア vs PdM のエスカレーション経路

## 既存 Figma DS からの移行

「Figma をデザインマスタにしない」（→ [05-figma.md](./05-figma.md)）と言ったとき、**既存ライブラリ・画面資産をいつまでどう残すか**。

- 段階的移行のマイルストーン（プレイグラウンド化のみ先に、など）
- 経営・他部署向け説明は Figma のまま続けるか、registry カタログを正とするか
- Figma ライセンス・運用コストを下げる前提か（下げないなら「マスタ不要」の意味が現場に伝わりにくい）

## Registry 未掲載の要望

カタログにない UI 要望は、OOUI の語彙に落ちるまで **従来どおり「絵」扱い** になりやすい。

- 新規 block 化の入口（誰が起票し、どの粒度で登録するか）
- 「載っていない = フロント未検証」と見なすか

## OOUI のドメイン前提

`Product` / `Cart` などのオブジェクト型は Shopify EC 前提が強い（→ [04-ooui.md](./04-ooui.md)）。

- 自社 API・別プラットフォームではオブジェクト辞書をどう揃えるか
- 商談中に API / DB の話まで踏み込む境界（顧客接点のスキル・責任範囲）

## 顧客向け未公認プロトの境界

営業 / CS / コンサルの **未公認デモ**（→ [08-stakeholder-workflow.md](./08-stakeholder-workflow.md)）が、契約・本番スコープと混同されないための線引き。

- お客さんに見せた案とリリース UI の対応の取り方（ブランチ / デモ URL 以外の運用）
- 未公認のまま本番に入った場合の責任分界
- 「イメージ提案」の明示テンプレ・チェックリストの要否

## デモデータと本番データ

fixtures / scenarios ベースの動く提案は、**本番のデータ構造・件数・エッジケース**とずれると炎上しうる。

- 商談デモで使うデータの出所と更新ルール
- 「デモで見えた挙動」と「本番で保証する範囲」の説明責任

## 還元ループの担い手

[09-workflow.md](./09-workflow.md) の「還元」まで、日常で誰がやるかが曖昧だと copy-in 文化が残る。

- プロダクト registry への再登録 PR を起こすデフォルト担当（エンジニア限定か、ロール横断か）
- 還元しない調整を許容するか、許すならいつまで

## 日常の単一入口

発見・提案・清書・掲載に使う **正のツール** が複数あると現場は従来フローに戻りやすい。

- カタログサイト / registry ブランチ / デモ URL / GitHub PR のどれを「公式の一覧」とするか
- 顧客接点と社内フローで同じ入口に揃えるか

## 顧客接点のイネーブルメント

公認 block 名・scenarios・AI 指示での compose は、**営業 / CS / コンサルがそのまま使えるとは限らない**。

- 最低限の研修・プロンプトテンプレ・禁止事項（fragment 改変など）の要否
- 壊した・誤解を生んだ場合のエスカレーション先
