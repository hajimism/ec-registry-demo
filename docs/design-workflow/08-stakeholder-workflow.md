# 顧客接点チームを含む基本フロー

[← 目次](./README.md)

## 概要

Registry はデザイン在庫の管理ではなく **デザイナーをつくる** 基盤である（→ [01-background.md](./01-background.md)）。生成 AI（→ [07-ai-enablement.md](./07-ai-enablement.md)）と組み合わせると、**お客さんとの接点を持つ営業・CS・コンサルタント** などが、自らプロトタイプを用いた UI 提案ができるようになる。

基本の流れ：

```
[営業 / CS / コンサル]
  公認 block + AI で顧客向けプロトタイプ提案
        ↓ 意図・構成・「こうしたい」が可視化された状態
[デザイナー]
  憲法に沿って清書・公認
        ↓
[エンジニア]
  fragment 契約を保った実装・プロダクト registry 登録・リリース
```

Registry の価値は「デザインの完全な民主化」ではなく、**提案の入口を顧客接点まで下げ、清書とリリースをプロに集約する** ことにある。品質の分散を抑えつつ、提案のスループットを上げる。

## 従来との違い（顧客接点）

| 従来 | Registry + AI |
|---|---|
| デザイナー待ちで提案が遅い | 商談・ヒアリング中に compose 案を出せる |
| 言葉だけの要件 | **動く・見える** 提案（scenarios 付きプレビュー） |
| 個人の PPT スキルに依存 | 公認部品ベースで品質の下限が担保 |

## 各ロールの担当

| ロール | 段階 | 担当 |
|---|---|---|
| **営業 / CS / コンサル** | 提案 | 公認 block の compose、AI による局所カスタム、顧客との合意形成。**未公認で可** |
| **デザイナー** | 清書 | トーン調整、ブランド、公認、AI の編集境界の維持 |
| **エンジニア** | リリース | fragment・API・DB の整合、プロダクト registry への還元、`shadcn add` 可能な状態 |
| **PdM** | 仕様 | 公認 block を仕様上の選択肢として利用 |

デザイナーは「最初の 1 枚」から解放され、**顧客と合意済みの案をプロダクト品質に落とす** に集中する。

## 公認ステージとの対応

| 段階 | 公認 | 用途 |
|---|---|---|
| 顧客提案用プロト | 未公認・WIP | 営業 / CS / コンサル + AI |
| 社内清書 | レビュー中 | デザイナー |
| 本番 | 公認 | エンジニアが add・デプロイ |

「お客さんに見せた案」と「リリースした UI」の対応を取るため、提案は **プロダクト registry のブランチ or デモ URL** に寄せて管理するのが望ましい（→ [06-governance.md](./06-governance.md)）。

## 顧客向けデモの注意

- 商談用は **イメージ提案** であることを明示する場合がある（トークン未確定など）
- 公認 block だけを使えば、見た目・挙動の下限は担保される
- データの話は OOUI の語彙で行う（→ [04-ooui.md](./04-ooui.md)）  
  例：「これは `Product` の既存フィールドで足りる」「メタフィールド追加が要りそう」

## 社内の技術ワークフローとの関係

エンジニア・PdM 向けの **発見 → 評価 → 採用 → 調整 → 還元** は [09-workflow.md](./09-workflow.md) を参照。顧客接点フローはその上流に位置し、**清書前の提案** を供給する。
