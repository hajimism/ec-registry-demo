# ガバナンス：デザイナーのお墨付き

[← 目次](./README.md)

## 基本方針

「好き勝手登録してよい」一方で、**プロダクト registry 公認デザイナーのレビューを受けた実装**にはマークを付ける。要は **デザイナーのお墨付きがあるかどうか** の話になる。

| 区分 | 意味 |
|---|---|
| 未公認 | 個人・チームの実験、WIP、フォーク試作、**顧客向けプロト**。登録自体は自由 |
| 公認 | トークン準拠・ブランドガイド・fragment 契約・scenarios 網羅・a11y などをレビュー済み |

イメージは npm の unscoped パッケージ vs verified publisher に近い。

## 提案段階との対応

[08-stakeholder-workflow.md](./08-stakeholder-workflow.md) のフローと対応づける：

| 段階 | 公認 | 主な関与者 |
|---|---|---|
| 顧客提案用プロト | 未公認 | 営業 / CS / コンサル（+ AI） |
| 社内清書 | レビュー中 | デザイナー |
| 本番リリース | 公認 | エンジニア（add・デプロイ） |

PdM / エンジニアは **公認マーク付き**を仕様・本番実装の選択肢にできる。未公認は探索・商談用に留める。

## 運用イメージ

- manifest（`registry.json` の item）に `certified` や `reviewedBy` などのメタを載せられる
- カタログ UI でバッジ表示、CLI では `--only-certified` のような opt-in もあり得る
- 提案の追跡はプロダクト registry のブランチ or デモ URL に寄せる

## デザイナーの役割の再定義

公認は「品質バッジ」だけではなく、**デザイナーをつくる** ランタイムのゲートである — 提案は広く許し、本番に載せる判断だけをプロが担う（→ [01-background.md](./01-background.md)）。デザイン**作成**がボトルネックだった時代より、**清書・公認** に重心が移る。

| | |
|---|---|
| ❌ | 全コンポーネントのピクセル管理、すべての提案の最初の 1 枚 |
| ✅ | プロダクト registry の **公認基準** と **レビューキュー** |
| ✅ | 顧客接点から上がってきた案の **清書** と公認 |
| ✅ | AI が触っていい境界の定義（→ [07-ai-enablement.md](./07-ai-enablement.md)） |
| ✅ | Figma プレイグラウンドでの試作の採用 / 却下（→ [05-figma.md](./05-figma.md)） |

エンジニアは未公認でも登録可能 → デザイナーが公認 → 「このプロダクトで使っていい部品」になる。
